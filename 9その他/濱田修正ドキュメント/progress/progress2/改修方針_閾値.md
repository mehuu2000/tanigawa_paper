## 最終目標
- 誤植なし正例データと負例データを使ってBM25, BM25正規化, RC, CC, MCの閾値を更新し、それを利用して再評価する

## 想定手順
※ 実作業の順番は `Step1 → Step3 → Step2 → Step4 → Step5` とする。  
理由は、Crossref を除いた JaLC-only の環境を先に構築してから誤植なし予備実験用データを生成した方が、その後の閾値再取得・再評価との整合性が取りやすいため。

#### 1, 論文のMCの完全再現
・ 論文内では " 0.8623 " という閾値を示している。(誤植あり正例データ+負例データ)
- しかし、論文作成者から入手したこのシステムにはMCの閾値を検証・実験するコードが存在していない。
- そのため、閾値の更新の前に誤植ありデータと負例データを利用してMCの閾値を実験するコードを作成する。
- そして、論文内の0.8623と一致するシステムを入手する。

#### 2, 誤植なし予備実験用データ(実在)の作成
・ 予備実験用の実在データも、現状は評価用の実在データと同様に誤植ありで生成されている。
- そのため、現在の `preliminary_experiment.ipynb` のままでは、誤植あり実在データ + 既存の予備実験用負例データで閾値を最適化することになる。
- 今回は誤植なしでの再評価を目的としているため、閾値取得に用いる予備実験用の実在データも誤植なしに揃える必要がある。
- なので、評価用実在データを誤植なしで作成した時と同様に、予備実験用実在データについても誤植なし版を新たに作成する。

#### 3, 環境構築
・ DB(MongoDB, Apache Solr)に登録しているデータに不要なデータが入っているため、登録し直す。
- 現状はJaLCデータとCrossrefデータどちらもDB内に入ってしまっている。(このシステムでの登録数の数とDBの登録数が一致しているため間違いはないはず)。
- そのため、DOIが重複していることがある。
- なので、Crossrefデータを抜いたJaLCデータだけが登録されているDBを再構築する。

#### 4, 閾値の再取得
・ 誤植なしデータと負例データを利用し、新しいBM25, BM25正規化, RC, CC, MCの閾値を取得する。
- 現状のBM25, BM25正規化, RC, CC, MCの閾値は過去の誤植ありデータ込みで最適化されている。
- そのため、誤植なしデータに置き換わった今、誤植なしデータを利用し、それに最適化された閾値を入手する必要がある。

#### 5, 再評価
・ 4で新しく取得した閾値を使用して再度このシステムの評価を出す。
- 今までのものは過去の閾値を利用しての評価だった。
- そのため、新しく入手した閾値を利用して、再度データからこのシステムの評価を出す。

## Step1 具体方針

### 目的
- 論文内に記載されている MC の閾値 `0.8623` を、このシステム上で再現する。
- ここでは再評価までは行わず、まずは「MC の閾値決定処理」を再現することを目的とする。

### 使用するデータ
- 正例データ: `5-1予備実験用データ/5-1-1参考文献文字列（実在）/positive_reference_*_pre.txt`
- 負例データ: `5-1予備実験用データ/5-1-2参考文献文字列（架空）/negative_reference_*_pre.txt`

### 使用する notebook
- `4プログラム/preliminary_experiment.ipynb`
- 閾値や全文書数などの共通設定は `4プログラム/config.py` で管理する。
- この notebook には RC / CC / BM25 / BM25正規化 の閾値決定処理は既に存在している。
- 一方で、MC の閾値決定処理は存在していないため、新たに追加する必要がある。

### 修正方針
- `preliminary_experiment.ipynb` の既存コードは変更しない。
- 既存の RC / CC / BM25 閾値決定セルの後ろに、MC 用のセルを追加する形で対応する。
- つまり、既存セルの書き換えではなく「追加のみ」で修正する。

### 実施内容
1. `preliminary_experiment.ipynb` の既存ロジックを踏襲して、MC 用のスコア算出セルを追加する。
2. 各参考文献文字列をクエリとして Apache Solr に検索をかけ、上位 10 件の候補文献を取得する。
3. 取得した各候補文献に対して、論文に記載されている式
   `MC = 2RC・CC / (RC + CC)`
   により MC スコアを計算する。
4. 参考文献文字列 1 件につき、候補 10 件の中で最も MC が高い候補のスコアを採用する。
5. これを正例データ・負例データそれぞれについて行い、MC スコアの一覧を作成する。
6. RC / CC / BM25 と同様に、正例・負例のスコア一覧を用いて F1 が最大となる閾値を求める。

### 確認内容
- 得られた MC の閾値が、論文記載の `0.8623` と完全一致するかを確認する。
- 一致判定のロジックは notebook 側には実装せず、結果を見て目視で確認する。

### 完了条件
- `preliminary_experiment.ipynb` 上で MC の閾値決定処理を再現できること。
- その結果として得られた MC 閾値が `0.8623` と完全一致すること。

## Step2 具体方針

### 実施順メモ
- ドキュメント上の番号は `Step2` だが、実作業では `Step3` の環境再構築後に実施する。
- これは、Crossref を除いた JaLC-only の MongoDB / Solr 環境を先に整え、その状態を参照して誤植なし予備実験用データを生成するためである。

### 目的
- `preliminary_experiment.ipynb` で使用する予備実験用の実在データを、誤植なし版に置き換える。
- ただし、予備実験用データの母集団は変更せず、差分を「誤植の有無」のみに限定する。

### 使用するデータ
- 基準 DOI リスト: `5-1予備実験用データ/5-1-1参考文献文字列（実在）/doi_list.txt`
- 参照元データ: MongoDB 上の JaLC 論文データ

### 出力先
- 新しい出力先ディレクトリとして
  `5-1予備実験用データ/5-1-1参考文献文字列_誤植なし（実在）`
  を用意する。
- このディレクトリには、基準となる `doi_list.txt` も複製して配置し、生成処理はこの複製済み `doi_list.txt` を参照する。

### 出力ファイル名
- `positive_reference_ipsj_pre_none_typo.txt`
- `positive_reference_jsai_pre_none_typo.txt`
- `positive_reference_lsj_pre_none_typo.txt`

### 使用する notebook
- `4プログラム/create_references.ipynb` は変更しない。
- `4プログラム/create_references_none_typo.ipynb` を利用して対応する。
- 既存の「評価用誤植なし正例データ作成」部分は変更せず、予備実験用実在データ作成のためのセルを追加または修正して対応する。

### 修正方針
- 評価用誤植なし正例データ作成時と同じ考え方で進める。
- 論文集合は `doi_list.txt` によって固定し、MongoDB を全走査して再抽出することはしない。
- これにより、既存の予備実験用実在データとの差分を「誤植あり / 誤植なし」だけに保つ。

### 実施内容
1. 元の `5-1-1参考文献文字列（実在）/doi_list.txt` を、新しい
   `5-1-1参考文献文字列_誤植なし（実在）`
   ディレクトリへ複製する。
2. `create_references_none_typo.ipynb` に、予備実験用実在データ作成セルを追加または修正する。
3. 生成処理では、新ディレクトリ側の `doi_list.txt` を読み込む。
4. `doi_list.txt` に記載された DOI を 1 件ずつ MongoDB に問い合わせ、対象論文データを取得する。
5. 取得した論文データを、誤植なしのまま JSAI / IPSJ / LSJ の 3 スタイルへ整形する。
6. 整形した結果を、新ディレクトリに `*_pre_none_typo.txt` として書き出す。

### 確認内容
- 生成された 3 つの `*_pre_none_typo.txt` が存在すること。
- 生成処理が `doi_list.txt` 基準で実行され、母集団が固定されていること。
- 既存の誤植あり予備実験用実在データとの差分が、誤植の有無のみになっていること。

### 完了条件
- 予備実験用実在データの誤植なし版を新ディレクトリへ生成できること。
- そのデータを、後続の `preliminary_experiment.ipynb` における閾値再取得で利用できる状態にすること。

## Step3 具体方針

### 実施順メモ
- 実作業では `Step1` の直後にこの `Step3` を先行して実施する。
- その後、再構築済みの JaLC-only 環境を用いて `Step2` の誤植なし予備実験用データ作成を行う。

### 目的
- 2023年の JaLC データのみが登録された MongoDB / Apache Solr 環境を再構築する。
- 現在混在している Crossref データを除外し、後続の閾値再取得と再評価を JaLC データのみで行える状態にする。

### 対象データ
- 2023年の JaLC データのみを対象とする。
- Crossref データは登録対象に含めない。

### 対象環境
- MongoDB: `jalc.restapi`
- Apache Solr: `jalc` core

### 前提
- 今回は core や schema を作り直すのではなく、内部データを削除して再登録する。
- MongoDB では collection を削除し、Solr では登録済みドキュメントを全件削除する。
- Solr の `solr.delete(q='*:*')` は登録済みドキュメントの全削除であり、core 自体や schema / field / copyField 設定は削除しない前提で進める。

### 実施内容
1. MongoDB の `jalc.restapi` collection を削除する。
2. 2023年 JaLC データのみを MongoDB に再登録する。
3. MongoDB 登録後、`doi` インデックスを付与する。
4. Solr `jalc` core に対して `solr.delete(q='*:*')` を実行し、登録済みドキュメントを全件削除する。
5. MongoDB に登録された 2023年 JaLC データを、Solr `jalc` core に再登録する。

### 確認内容
- MongoDB の再登録が完了していること。
- Solr の全件削除が完了していること。
- Solr への再登録が完了していること。
- 再登録後、検索処理が正常に動作すること。

### 完了条件
- MongoDB に 2023年 JaLC データのみが登録された状態になること。
- Solr `jalc` core に 2023年 JaLC データのみが登録された状態になること。
- 後続の `preliminary_experiment.ipynb` および `evaluation_experiment.ipynb` を実行できる環境が整うこと。

- 対象は `BM25` `BM25正規化` `RC` `CC` `MC` の5指標とする。

### 使用する notebook

### 使用するデータ
- 正例データ:
  - `positive_reference_ipsj_pre_none_typo.txt`
  - `positive_reference_jsai_pre_none_typo.txt`
  - `positive_reference_lsj_pre_none_typo.txt`
- 負例データ:
  - 既存の `negative_reference_*_pre.txt`

### 修正方針
- `preliminary_experiment.ipynb` の入力ファイルパスを、誤植なし予備実験用実在データに直接置き換えて実行する。
- 今回は既存セルを残したまま追加するのではなく、ファイルパスを直接変更する前提で進める。
- 負例側の入力ファイルパスは既存のままとする。
- 閾値の管理は notebook 内のマジックナンバーではなく `config.py` を利用して行う。

### 実施内容
1. `preliminary_experiment.ipynb` の正例入力パスを、誤植なし予備実験用実在データへ変更する。
2. 負例入力には、既存の予備実験用負例データをそのまま利用する。

4. 正例・負例のスコア分布を用いて、各指標ごとに F1 が最大となる閾値を求める。
5. 取得した閾値を記録する。
6. 取得した閾値を `config.py` の current values に反映し、以前の値はコメントブロックとして残す。

### 確認内容
- 5 指標すべてについて、閾値が取得できていること。
- 正例には誤植なし予備実験用実在データが、負例には既存の予備実験用負例データが使われていること。

### 完了条件
- `BM25` `BM25正規化` `RC` `CC` `MC` の新しい閾値を取得できること。
- 取得した閾値を、後続の再評価で利用できる状態にすること。

## Step5 具体方針

### 目的
- `Step4` で取得した新しい閾値を利用して、このシステムを再評価する。
- 誤植なし正例データと既存の評価用負例データを用いて、各手法の評価結果を再取得する。

### 使用する notebook
- `4プログラム/evaluation_experiment.ipynb`
- 閾値や全文書数などの共通設定は `4プログラム/config.py` で管理する。

### 使用するデータ
- 正例データ:
  - `positive_reference_ipsj_eval_none_typo.txt`
  - `positive_reference_jsai_eval_none_typo.txt`
  - `positive_reference_lsj_eval_none_typo.txt`
- 負例データ:
  - 既存の `negative_reference_*_eval.txt`

### 対象手法
- `BM25`
- `BM25正規化`
- `RC`
- `CC`
- `MC`

### 修正方針
- `evaluation_experiment.ipynb` の入力ファイルパスを、誤植なし正例データに直接置き換えて実行する。
- 負例側の入力ファイルパスは既存の評価用負例データのままとする。
- 各手法の閾値は、`Step4` で取得した新しい値を `config.py` に反映し、その値を notebook から参照する。

### 実施内容

4. 正例評価を実行し、`case1` `case2` `case3` `case4` の結果を取得する。
5. 負例評価を実行し、`True negative` `False positive` の結果を取得する。
6. 結果をスタイル別および合計値で整理する。

### 確認内容
- 5 指標すべてについて、正例・負例の評価結果が取得できていること。
- 正例には誤植なし正例データが、負例には既存の評価用負例データが使われていること。
- 各手法の閾値が、`Step4` で取得した新しい値として `config.py` に反映されていること。

### 完了条件
- `BM25` `BM25正規化` `RC` `CC` `MC` の新しい閾値で、正例・負例の再評価結果を取得できること。
